# 三阶段模拟用研方法论 — 原理与示例

> 本文档解释方法论背后的「为什么」，提供正反例、边界案例和深入分析。具体执行步骤请参阅 SKILL.md。

## 为什么需要这个方法

### 传统模拟用研的缺陷

当我们用 AI 模拟用户角色时，最大的陷阱是**预设污染**：

```
角色 Prompt 中写了 → 角色当然会说出来 → 研究者以为是"发现" → 实际是自我验证
```

典型例子：
- 在角色 Prompt 中写「经常觉得信息散落在多个工具里」→ 角色果然抱怨信息分散 → 研究者认为这是"真痛点"
- 但如果不预设这一条，角色可能根本不会提到这个问题，他的真实困扰可能完全不同

### 三阶段法的核心逻辑

```
Phase 1: 只给灵魂，不给痛点 → 痛点从内在逻辑自然生长
Phase 2: 逐角色提取 + 交叉分析 + JTBD 提炼 → 多层次理解用户
Phase 3: 带着自己的故事碰撞产品 → 反应基于真实经历而非预设态度
验证关卡: 四个独立审计 agent → 确保每个环节的产出质量
```

## 角色设定的黄金原则

### 应该定义的（灵魂）

| 维度 | 说明 | 为什么需要 |
|------|------|---------|
| 人口统计 | 年龄、性别、城市 | 基础锚定 |
| 职业背景 | 行业、职位、从业年限 | 决定视角和经验深度 |
| 工作规模 | 日常并行任务量、协作人数 | 决定日常压力和工作模式 |
| 性格标签 | 3-5 个关键词 | 决定说话方式和反应模式 |
| 工具栈 | 日常使用的工具和系统 | 决定技术素养和迁移成本 |
| 认知倾向 | 对新事物的接受方式 | 决定对新工具的接受度 |

### 绝不应该定义的

| 维度 | 反例 | 为什么不能写 |
|------|------|-----------|
| 具体痛点 | 「经常觉得效率低下」 | 这应该让角色自己发现 |
| 痛点案例 | 「上周因为 X 问题浪费了半天」 | 这是预设的例子，不是角色自己的经历 |
| 对产品的态度 | 「觉得现有工具不够智能」 | 应该让角色在碰撞时自然形成态度 |
| 付费意愿 | 「愿意花 500/月」 | 这应该在 Phase 3 中自然表达 |
| 功能需求 | 「最需要自动化报告功能」 | 这完全是预设结论 |

### 判断标准与灰色地带

核心问题：**如果这个人是真实存在的，他的朋友会不会用这些词来描述他？**

- 「她性格犀利，说话直接」→ 朋友会这么说 ✅
- 「她觉得现有流程中有三个瓶颈需要优化」→ 朋友不会用这种精确的工作评价来描述她 ❌

**灰色地带示例**——有些描述看起来像灵魂，实际暗含了痛点预设：

| 描述 | 判定 | 原因 |
|------|------|------|
| 「她经常加班到很晚」 | ⚠️ 看语境 | 如果是性格特征（工作狂），可以保留；如果暗示「工作效率低所以加班」，则是预设痛点 |
| 「他管理一个 8 人的技术团队」 | ✅ | 纯客观描述 |
| 「他管理一个经常延期的 8 人团队」 | ❌ | 「经常延期」是预设的痛点 |
| 「她每天要看 200 封邮件」 | ✅ | 客观描述工作量，痛不痛由角色自己说 |
| 「她被 200 封邮件淹没」 | ❌ | 「淹没」预设了这是个负面体验 |

## 角色组合设计

### 为什么需要组合设计

单个角色的设定再好，如果一组角色全是同质的（同一个城市、同一个资历层、同一种技术接受度），研究结果就只代表一个切面的声音。组合设计确保不同视角之间产生碰撞和张力。

### 差异维度的原理

**资历梯度**是最重要的差异轴——新手的痛点往往是「不知道该怎么做」，中段是「知道该怎么做但流程太低效」，资深是「流程可以但缺乏战略视角或创新空间」。如果所有角色都在同一个资历层，你只能看到一种类型的痛点。

**组织规模**决定了工作模式的根本差异——个人/自由职业者的痛点是「什么都得自己做」，小团队是「协作成本高但灵活」，大机构是「流程成本高但有资源」。同样的功能在不同组织规模下的价值感知可能完全不同。

**技术接受度**影响产品碰撞的反应模式——保守型关注可靠性和迁移成本，尝鲜型关注新功能和可能性。如果所有角色都是尝鲜型，你会高估产品的接受度。

### 快速自检方法

想象这组角色坐在同一张圆桌上讨论同一个话题：

- **好的组合**：一个工作两年的年轻人说「我觉得 AI 工具挺好用的」，一个十五年的老前辈说「我见过太多号称颠覆行业的工具了」——他们会争论起来
- **差的组合**：八个人都是一线城市、5-8 年经验、对新技术持开放态度——他们会频频点头，研究也只是在回声室里打转

### 对比示例

**同质化组合**（问题示范）：

| 角色 | 城市 | 资历 | 组织 | 技术接受度 |
|------|------|------|------|-----------|
| P1 | 上海 | 6 年 | 中型公司 | 开放 |
| P2 | 北京 | 7 年 | 中型公司 | 开放 |
| P3 | 深圳 | 5 年 | 中型公司 | 中立 |
| P4 | 杭州 | 8 年 | 大公司 | 开放 |

问题：全是一线城市、全是中段资历、组织规模差异不够、技术接受度偏高。

**多样化组合**（推荐示范）：

| 角色 | 城市 | 资历 | 组织 | 技术接受度 |
|------|------|------|------|-----------|
| P1 | 上海 | 1 年 | 大公司 | 尝鲜 |
| P2 | 成都 | 10 年 | 自由职业 | 保守 |
| P3 | 北京 | 4 年 | 创业团队 | 开放 |
| P4 | 长沙 | 15 年 | 大机构 | 中立 |

## Phase 1 叙事节奏设计

### 五段式节奏的原理

| 段落 | 目的 | 产出 |
|------|------|------|
| 热身 | 让角色进入自己的语境 | 当前状态、情绪基调 |
| 日常流 | 还原完整工作流 | 信息流动路径、工具切换、协作关系 |
| 深潜 | 挖掘具体案例 | 一挫败一得意，有时间线和情绪 |
| 远望 | 拉高视角 | 行业观察、自我定位 |
| 许愿 | 提炼核心渴望 | 最想改变的一件事及原因 |

**为什么用节奏而非问题清单**：问题清单会让角色逐条回答，产出像问卷。节奏引导让角色自然从一个话题流向下一个，叙事更连贯，痛点出现得更自然。

### 好的引导 vs 坏的引导

好的引导（开放、不引导方向）：
```
- 最近在忙什么？心情怎么样？
- 一天是怎么过的？信息怎么流动？
- 最近一个让你挫败的具体案例
- 最近一个让你得意的具体案例
- 对行业/职业的观察
- 如果有魔法棒，你最想改变什么？
```

共同特点：不预设方向、要求具体案例、覆盖正面和负面。

坏的引导（引导性太强）：
```
❌ 你在 XX 流程中遇到什么困难？（预设了特定流程是痛点领域）
❌ 你觉得 AI 能帮你做什么？（引导角色思考 AI 的用途）
❌ 你对现有工具的效率满意吗？（暗示效率是问题）
```

## Phase 2 分析框架

### 两步分析的原理

Phase 2 分为逐角色提取（2a）和交叉综合（2b）两步，原因是：

1. **Context 压力**：8 角色 × 3-4k 字 = 24-32k 字的原始叙事，一个 agent 读完再做三层分析，后面的层容易分析变浅
2. **信息衰减**：LLM 处理长文本时，后面的内容对前面内容的记忆会衰减。先结构化提取再综合，每一步的输入量可控（< 6k 字）
3. **并行加速**：Phase 2a 的逐角色提取可以并行执行，缩短总耗时

### 旅程地图方法

从每个角色的「日常流」叙述中提取工作流节点，形成旅程地图：

```
节点 A → 节点 B → 节点 C → 节点 D → ...
  😐        😤        😐        😊
```

每个节点标注：
- **做什么**：角色在这个环节的具体动作
- **用什么**：工具、系统、沟通渠道
- **和谁**：协作对象
- **情绪**：正面/中性/负面，以及原话证据

然后**叠加所有角色的旅程**，找出痛点聚集的流程位置。这比单纯列痛点更有价值——它告诉产品团队应该在工作流的**哪个环节**介入。

### 痛点分类体系

| 类型 | 定义 | 判断示例 | 行动 |
|------|------|---------|------|
| 共性强信号 | N/M 角色自然提到（N > M/2） | 6/8 角色都在日常流中描述了「在多个工具间反复切换寻找信息」，且都伴随负面情绪 | 高优先级产品方向 |
| 分众信号 | 特定 1-2 个角色强烈但不普遍 | 只有自由职业的 P2 强烈表达了「没有人帮我 review」的孤独感 | 特定用户群的差异化价值 |
| 假设验证 | 预设的假设在自由叙事中也出现 | 预设「信息碎片化是痛点」，P1、P3、P5 在不知道这个假设的情况下都描述了类似困扰 | 确认为真，保持优先级 |
| 可疑信号 | 预设的假设在自由叙事中未出现 | 预设「用户需要自动报告生成」，但 8 个角色没有一个自发提到报告是痛点 | 降低优先级或重新验证 |
| 新发现 | 未预设但自由叙事中自然出现 | 3/8 角色不约而同地描述了「给新人解释上下文很痛苦」——这不在假设清单中 | 重点关注，可能是被忽略的真机会 |

### 分析纪律

**精确量化**——所有统计表述用 "N/M" 格式。

```
✅ "6/8 角色在描述日常工作时自发提到了信息碎片化"
❌ "大多数角色都提到了信息碎片化的问题"
```

为什么：模糊词让不同读者脑中的数字不同（"大多数"可以是 5/8 也可以是 7/8），精确数字消除歧义。

**观察与解读分离**——每个痛点分两层记录。

```
📋 观察：P2 在描述挫败案例时说"我花了一个半小时从四个群里翻聊天记录，最后拼出来的东西还是漏了两条关键信息"
💡 解读：信息散落在多个沟通渠道中，手动归集既耗时又不可靠，存在关键信息遗漏风险
```

为什么：观察是事实，不可争辩；解读是判断，可能有误。区分二者让读者能自行验证你的推理。

**引用带 ID**——所有原话摘录标注角色 ID（P1-Pn），确保可溯源。

为什么：没有 ID 的引用无法验证——读者不知道这话是哪个角色说的，也无法回到原文查看上下文。

### 外部数据交叉验证

如果用户提供了真实用研数据（访谈记录、工单、NPS、评论等），在痛点聚类完成后增加交叉验证：

| 交叉状态 | 含义 | 信号强度 |
|---------|------|---------|
| 模拟 + 真实双重出现 | 模拟角色和真实用户都提到 | 最强——高度可信 |
| 仅模拟出现 | 模拟角色提到但真实数据中未出现 | 需谨慎——可能是模拟偏差，也可能是真实用研未覆盖 |
| 仅真实数据出现 | 真实用户提到但模拟角色未生成 | 需检查——角色设定是否有盲区 |

交叉验证的价值在于：模拟数据擅长发现「完整的叙事和情绪脉络」，真实数据擅长确认「这个问题在现实中真的存在」。二者互补，而非互相替代。

### JTBD 提炼方法

Jobs to be Done 把痛点从「症状」提升为「需求」——用户不是在抱怨，他是在试图完成一个任务但受阻了。

**提炼路径**：
1. 从痛点出发：角色在抱怨什么？背后他想完成的任务是什么？
2. 从得意案例出发：角色在什么时候感到成就感？他完成了什么 job？
3. 从许愿出发：魔法棒问题的答案直接指向最重要的未满足 job

**正例**：
> 当我在做竞品分析、需要快速了解一个新领域时，我想要高效地把碎片信息结构化，这样我就能在一天内形成有说服力的判断而非花三天翻资料。
> 来源：P1（"一天翻了五六十个网页"）、P4（"老板给我两天时间搞清楚这个市场"）
> 当前替代：手动收藏网页 + 开 Notion 整理 + 找行业朋友口头问
> 不足：手动整理效率低、依赖个人判断可能有盲区、时间压力下容易遗漏

**反例**（过度抽象）：
> ❌ 当我在工作时，我想要更高的效率，这样我就能做更多事。
> 问题：情境太泛、目标太泛、结果太泛——这适用于地球上所有打工人，没有任何指导价值

JTBD 的价值在于它是**产品语言**——产品团队可以直接用它来定义功能范围和价值主张。

## Phase 3 产品描述原则

### 中性描述 vs 营销语言

```
❌ 营销语言：「XX 帮你从海量信息中提炼精准洞察，让你的决策效率翻倍」
✅ 中性描述：「你把相关内容倒进去，系统从中提取结构化信息。积累够了你点一个按钮，它生成一份摘要。」
```

原则：
- 只描述**功能事实**（输入什么、输出什么、怎么操作）
- 不描述**价值判断**（更快、更准、更好）
- 不使用**情感词汇**（智能、强大、颠覆）
- 让角色**自己判断**这个东西对他有没有用

### 价值感知探针的原理

Phase 3 中加入定价/价值感知探针，原因是：

1. **商业化决策依据**：知道角色觉得产品有用不够，还需要知道他们觉得值多少钱、从什么预算出
2. **替代方案锚定**：角色的付费意愿不是凭空产生的，而是参照他们现有替代方案的成本（时间成本、金钱成本）
3. **分群差异**：不同职业层次、组织规模的角色对价格的敏感度和预算来源完全不同

**注意**：AI 模拟角色的付费数字**不能直接作为定价依据**——它提供的是「哪些因素影响付费决策」「不同分群的预算逻辑」等定性洞察，定量定价需要真实市场验证。

## 多概念对比方法

### 为什么需要多概念对比

真实的 concept testing 经常需要对比方案——完整版 vs 轻量版、AI 主导 vs 人主导、SaaS vs 本地部署等。单概念碰撞只能告诉你「角色对这个概念怎么看」，多概念对比还能告诉你「角色在不同方案之间怎么选」。

### 对比的关键维度

对比不是简单地把两个概念的碰撞结果并排放——而是要分析：

1. **分群偏好**：哪个概念在哪个用户分群中更受欢迎？为什么？
2. **取舍逻辑**：角色在两个概念间做选择时，考虑的首要因素是什么？（成本、复杂度、灵活性、安全性……）
3. **互补空间**：两个概念是否可以互补而非互斥？在什么场景下用 A、什么场景下用 B？
4. **方向建议**：基于分群偏好和取舍逻辑，给出推荐方向

### 注意事项

- 每个概念必须分别用中性描述，不暗示哪个更好
- 同一角色对不同概念的碰撞必须在独立的 agent 中进行，避免前一个概念影响后一个的判断（锚定效应）
- 多概念模式复用与单概念相同的 Phase 1-2 产出——角色的故事和痛点不会因为测试了几个概念而改变

## 验证体系设计原理

### 为什么需要独立验证 Agent

整个 pipeline 存在四个关键的质量风险点：

| 关卡 | 风险 | 如果不检查会怎样 |
|------|------|----------------|
| V1（Phase 1 后） | 叙事太短/太泛/人设崩塌/预设污染 | Phase 2 分析空中楼阁，痛点不可信 |
| V2（Phase 2 后） | 逻辑跳跃/引用失真/遗漏信号 | 综合报告结论站不住脚 |
| V3（Phase 3 后） | 人设不一致/反应同质化/过度热情 | 碰撞反应失真，产品决策被误导 |
| V4（报告后） | 结论与证据脱节/章节缺失/数字不一致 | 研究报告不可信赖 |

每个环节的问题都会**向后传染**，越早发现越好。

### 为什么用独立 Context Window

验证 agent 使用独立的 context window（通过 Agent 工具启动），原因是：

1. **视角隔离**：主流程 agent 已经「看过」了前面的内容，有确认偏误的风险——它倾向于认为自己产出的东西是好的。独立 agent 带着全新的视角审视产出。
2. **标准明确**：独立 agent 只拿到产出和评判标准，不受「我知道为什么写成这样」的心理影响。
3. **可追溯**：每个验证 agent 产出独立的验证报告（v1-v4-validation.md），这些报告本身也是研究质量的证据。

### PASS / FLAG / BLOCK 三级判定

- **PASS**：质量达标，进入下一阶段
- **FLAG**：有问题但不致命，记录在案。FLAG 的问题会在综合报告的「局限性」和「知识缺口」中诚实说明
- **BLOCK**：有严重问题，必须修复后才能继续。每个 BLOCK 判定必须附带**具体的改进方向**（重做什么、怎么改进），确保重做有方向而非盲目重试

### 验证报告的价值

四份验证报告（v1-v4-validation.md）不是流程产物——它们是研究质量的元数据。读者可以通过验证报告了解：
- 这份研究经历了怎样的质量把关
- 哪些环节有 FLAG（即研究团队意识到但未能解决的问题）
- 研究结论的可信度边界在哪里

## 综合报告结构

### Executive Summary 写作原则

报告的第一个部分，3-5 句话，回答三个问题：
1. 最重要的发现是什么？
2. 对产品意味着什么？
3. 最大的不确定性在哪？

写法：结论先行，不要铺垫。读者看完这段就应该知道要不要继续读完整报告。

### 用户分群表

从碰撞反应中自然浮现的用户群体，用结构化表格呈现：

| 分群 | 特征 | 核心需求 | 覆盖角色 | 采用信号强度 |
|------|------|---------|---------|-----------|
| {名称} | {职业特征、行为特征} | {最重要的 JTBD} | P1, P4, P6 | 强/中/弱 |

分群的价值在于：不同群体需要不同的产品入口、不同的价值主张、不同的营销策略。

### 知识缺口清单

不是笼统的"需要进一步研究"，而是具体的问题列表：

```
1. {具体问题} — 建议方法：{访谈/问卷/A-B test} — 建议对象：{什么样的用户}
2. ...
```

每个缺口标注：为什么本次研究没有回答（角色设定不覆盖？提问没触及？模拟固有局限？验证关卡 FLAG 了哪些相关问题？）

## Impact/Effort 矩阵

功能优先级不能只看痛点强度，还要考虑实施成本。用二维矩阵排列：

```
        高影响力
           │
    ┌──────┼──────┐
    │ 快赢  │ 大赌注│
    │Quick │ Big  │
    │ Win  │ Bet  │
    ├──────┼──────┤
    │ 填充物│ 陷阱  │
    │Fill  │Money │
    │      │ Pit  │
    └──────┼──────┘
           │
        低影响力
    低成本 ←──→ 高成本
```

- **Quick Win**（高影响 + 低成本）：立即做
- **Big Bet**（高影响 + 高成本）：值得投资，但要规划
- **Fill**（低影响 + 低成本）：顺手做，不要专门排期
- **Money Pit**（低影响 + 高成本）：避免

影响力来自 Phase 2（痛点强度 × 覆盖角色数 × JTBD 重要度）。成本需要结合产品和技术团队的判断——综合报告中给出初步评估，但标注「需技术团队确认」。

## Highlight Reel 编制方法

Highlight Reel 不只是摘录金句，而是从全部素材中提取**最有洞察力的片段**，分四类：

| 类型 | 什么样的片段 | 用途 |
|------|------------|------|
| **关键行为** | 角色描述的非预期工作方式或变通方法 | 揭示真实工作流，启发功能设计 |
| **决策转折** | 角色在碰撞产品时从犹豫变为感兴趣（或反过来）的瞬间 | 理解采用/放弃的触发点 |
| **情绪峰值** | 叙事中情绪最强烈的段落（无论正面负面） | 最真实的痛点和渴望所在 |
| **品牌素材** | 角色用自己的话描述需求的原话 | 可直接用于产品文案和营销 |

每条标注来源角色、出现阶段（Phase 1/3）、原文摘录。

## 方法局限性

以下局限必须在每份研究报告中诚实说明：

1. **仍是 AI 模拟**——再好的方法也不能替代真实用户访谈
2. **角色可能"太理性"**——真实用户受惯性、情绪、社交压力影响更大
3. **无法捕捉非言语信号**——犹豫、困惑表情、操作卡点
4. **适用于假设生成，不适用于假设验证**——本方法的产出是"更好的假设清单"，不是"结论"
5. **Impact/Effort 中的成本评估是粗略的**——影响力来自用研数据，但实施成本需要技术团队校准
6. **价值感知/定价数据仅供参考**——AI 模拟角色的付费意愿不能直接作为定价依据，需要真实市场验证
